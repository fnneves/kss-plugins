kss · overview

Keep Shit Simple

A solo-dev framework for Claude Code with reduced ceremony, durable state across sessions, and an opt-in self-improvement loop. Eleven small skills, a clean folder convention, and one rule: never auto-commit.

The problems it solves 01 The shape 02 How it improves itself 03 Where it fits 04

What's worth knowing up front

1. kss is built around durable state files, not chat memory. When you come back tomorrow, the state is on disk under .kss/ — not in your head, not in a context window.

2. The self-improvement loop is opt-in. You only invoke skill-autopsy when a skill misfires; nothing watches you in the background.

3. No skill ever runs git commit. Every artifact is a file you can review, edit, or delete before any commit happens. Trust is earned by being legible.

01

The problems it solves

3 pain points

Solo dev with an AI agent has a specific failure mode: every session starts cold, the agent re-asks what you already decided, and the work-in-flight gets reasoned about from a fresh slate. Heavier frameworks address this with rituals and roles. kss addresses it with files.

Problem 01

Cold-start every session

You closed the laptop yesterday mid-thought. Today the agent has no idea what you were doing, why, or what's next. You spend the first 10 minutes re-explaining.

→ start-session reads STATE.md + recent LOG.md entries and tells you exactly where you left off.

Problem 02

Multiple work tracks, one head

You have three half-finished initiatives. Without isolation, ideas bleed across. You commit to working on A, end up touching B, lose the thread on both.

→ topics isolate work tracks. milestones scope finite chunks inside a topic. Only one is active at a time.

Problem 03

Insights rot in chat history

You learn something useful in session 12. By session 30, neither you nor the agent remembers it. You re-discover the same gotcha quarterly.

→ capture parks ideas. distill promotes durable insights to CANONICAL-KB.md + domain terms to VOCABULARY.md.

Problem 04

Skills you can't fix

Most agent frameworks ship rigid skills. When one consistently misfires, you have nowhere to put the feedback — so the same friction recurs forever.

→ skill-autopsy turns a frustration into a report, then into a concrete diff against the offending SKILL.md.

Problem 05

Risky exploration, expensive cleanup

You want to test "what if". Branching feels heavy; commit feels premature. Without a parking spot, exploration leaks into the main work and pollutes the milestone.

→ spike creates a throwaway folder under .kss/spikes/ with one job: produce a verdict (keep / drop / promote).

The shape

11 skills

Skills are organized in five layers. You only invoke what the situation demands; there's no orchestrator running them in sequence.

Setup 2 skills

One-time bootstrap per project.

scaffold-project · map-codebase

Lifecycle 3 skills

Topic + milestone management.

new-topic · plan-milestone · complete-milestone

Session 2 skills

Bookend every coding session.

start-session · wrap-up

Knowledge 2 skills

Park ideas, distill durable insights.

capture · distill

Exploration 1 skill

Throwaway "what if" experiments with a verdict.

spike

Meta 1 skill

Self-improvement loop. Optional.

skill-autopsy

"Two-level state. .kss/STATE.md is the pointer. The active topic's STATE.md is the content."
03

How it improves itself

opt-in loop

skill-autopsy is the only entry point that captures friction with a skill and proposes a fix. It runs only when you call it — never automatically. The design treats reports as personal artifacts, never as plugin source.

Step 1 · Capture

One frustration becomes one report

You invoke skill-autopsy <skill> right after a misfire. A short interview captures: what you asked, what the skill did, why it was off. The report is appended (newest-on-top) to ~/kss-skill-reports/<skill>.md.

Reports live in your home folder. Never inside the plugin source — even forkers don't accidentally commit personal frustrations upstream.

Step 2 · Accumulate, with caps

Patterns need samples

One report is a complaint. Three reports might be a pattern. The autopsy stays silent through the first three; at four it offers to consolidate; past five it nudges harder. The cap exists so the file doesn't rot into a graveyard.

Step 3 · Propose, never invent

Concrete diff or "nothing yet"

Consolidate mode reads the reports and looks for recurring root causes. If there's signal — three reports all complain about the same missing preflight — it proposes a diff to that SKILL.md. If reports are vague, it says so honestly and suggests waiting.

Never fabricates. "Nothing to change yet" is a valid outcome.

Branch on source availability

forker / contributor If ~/.kss-source points at a writable clone, the diff is applied to that clone's SKILL.md. Reports are cleared. The skill prints the exact git -C commands to commit — but never runs them.

plain installer If no source path is set, the diff is printed for manual handling. Open a GitHub issue with it, fork later and apply it, or adapt your usage. Reports can be cleared either way.

04

Where it fits

positioning

kss is opinionated about what it isn't. It is not a startup-team simulator and not a deploy pipeline.

Heavyweight frameworks

gstack-style

  • ~45 skills modeling a startup team (CEO, Eng, Designer, QA, SRE)
  • Multi-step pipelines that ship code: /ship, /land-and-deploy, /canary
  • Optimized for ~10K LOC/week throughput
  • Auto-commits, auto-deploys, auto-monitors

kss

This framework

  • 11 skills; no roles, no orchestration
  • State lives in plain markdown files under .kss/
  • Optimized for continuity across many short sessions
  • Never auto-commits. Every artifact is a file you review

Inspired by GSD (Get Shit Done) — kss borrows the lifecycle ideas (topic isolation, codebase map, trigger-conditioned seeds) without the multi-agent ceremony. The "shit" pun is on the nose: keep the structure, drop the theatre.
